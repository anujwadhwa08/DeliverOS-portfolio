MCP Agent Ordering

How any AI agent places a real food order — with full security
Model Context Protocol Agent-to-Agent Production-Ready
MCP turns DeliverOS into a first-class citizen of the agentic web. A user's AI assistant — in Claude Desktop, a voice interface, or a custom agent — can search restaurants, browse menus, and place a real charge-to-card order, auto-creating an account on first use. Existing customers are protected by OTP verification so no agent can impersonate them without access to their inbox.

Step-by-Step Flow
1

Search Restaurants API Tool

The agent calls search_restaurants with a natural-language query: "find me a Thai place near Shoreditch under 30 min". The MCP server runs semantic vector search across the restaurant catalog — not a keyword match, but meaning-based search using OpenAI embeddings + Pinecone. Returns name, rating, ETA, and distance.

// Agent call
search_restaurants({ "query": "Thai food under 30 min Shoreditch", "lat": 51.52, "lon": -0.07 })
// Returns: [{ name, cuisine, eta_min, distance_km, rating }]
2

Browse the Menu API Tool

The agent fetches the full menu with get_menu — categories, items, descriptions, prices. The agent presents options conversationally: "They have Pad Thai for £12.50 and green curry for £13.00 — which would you like?". No hardcoded category lists or pagination logic needed by the agent.

3

Check Delivery Estimate API Tool

get_delivery_estimate returns ETA and delivery fee for a given restaurant and delivery address. The agent can confirm with the user before proceeding: "That's £2.50 delivery, arriving in ~28 minutes — shall I place the order?"

4

Identity Verification — Two Paths Security Gate

Before place_order can be called, the caller must provide a signed short-lived token. How they get it depends on whether they have an existing account.

🆕 New Customer — Auto-Onboard
  1. Agent calls place_order with newCustomer: true + email + explicit T&C consent flag
  2. MCP server creates a new customer account, saves email (SHA-256 hashed for lookup)
  3. Returns a signed 15-min JWT token — no OTP needed for first-time sign-up
  4. Agent retries place_order with the token — order is placed and card is charged
🔒 Existing Customer — OTP Gate
  1. Agent calls verify_agent_identity with the customer's email
  2. MCP server sends a 6-digit OTP to that email via Resend — 5 min expiry, single use
  3. User reads the code from their inbox and gives it to the agent
  4. Agent calls verify_agent_identity again with the OTP → receives signed 15-min JWT
  5. Agent retries place_order with the token — order placed, saved card charged
5

Place Order & Charge Card Auto-triggers restaurant + driver

place_order creates the order in Supabase, fires Stripe payment capture, and instantly notifies the restaurant's Telegram bot and the driver assignment queue. The agent gets back an order ID, ETA, and status. From this point the order proceeds through the same 20-state machine as any app or Telegram order.

// One call creates the order, charges the card, notifies restaurant + driver
place_order({ "restaurant_id": "r_abc123", "items": [{...}], "token": "eyJ..." })
// Returns: { order_id, status: "RestaurantNotified", eta_min: 28 }
6

Track Live Status API Tool

check_order_status returns the current order state, driver location, and updated ETA at any point. The agent can proactively update the user: "Your driver just picked up the order — 12 minutes away". Same data the mobile app tracks.

Why OTP for existing customers — and not for new ones?

The security model is intentional. A new customer has no saved cards or order history to steal — so the risk of auto-creating an account on their behalf is low, and the friction cost is high. We require explicit T&C consent and a valid email to create the account; no more trust than a standard sign-up form.

An existing customer has saved payment methods and a history of real transactions. If any agent can order as them just by knowing their email, a malicious or misconfigured agent could charge their card without permission. The OTP step ensures the real human was present at the moment of authorisation — the same guarantee a one-time password provides on a bank login. The 15-minute JWT expiry means even a captured token has a narrow window of use.
