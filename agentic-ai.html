Seven Agentic Patterns — What Each One Does
🤖
Pattern 1 — Architecture Multi-Agent Specialization

Three separate AI agents replace one generic assistant. Each is tuned to its user's context, vocabulary, and workflow. The customer agent handles ordering and tracking. The restaurant agent handles order confirmation, menu queries, and handover codes. The driver agent handles jobs, pickup, and earnings.

Why: a single generic AI gives worse answers to everyone. Specialisation means each agent stays in context, avoids confusion, and never leaks one user's information to another.

🔍
Pattern 2 — Retrieval RAG with Two-Tier Retrieval

Before any agent calls Claude, it runs a two-tier retrieval. Tier 1 (always): Supabase fetches all active restaurants and menus — nothing is silently missing. Tier 2 (query): Pinecone vector search finds the most semantically similar items using OpenAI embeddings.

Why: Claude can only reference what was retrieved. If a restaurant isn't in the context block, the agent cannot mention it — preventing confident hallucinations about menus or prices.

⚖️
Pattern 3 — Verification LLM-as-a-Judge

Every Sonnet response — across all three agents — is passed to Claude Haiku before reaching the user. The judge has one narrow task: check whether every restaurant name and menu item in the response actually appears in the retrieved context JSON. Any invented detail gets corrected before delivery.

Why Haiku, not a stronger model: this is a structured grounding check, not quality evaluation — essentially "does string X exist in list Y?" Haiku handles it in ~200ms at a fraction of Sonnet's cost. A larger judge would be overkill for a deterministic lookup task.

🔧
Pattern 4 — Action Agentic Tool Use

Agents don't just respond — they take real actions by calling structured tools. When a customer confirms an order, the agent: creates the order record in the database, authorises the payment via Stripe, notifies the restaurant, and starts the driver assignment workflow — all within a single conversational turn.

Why: information exchange alone isn't useful. Agentic tool use means one user message produces real-world outcomes.

🔒
Pattern 5 — Privacy PII-Aware Routing

A dedicated router layer runs before any agent. It intercepts messages containing names, addresses, or contact details — encrypts them with AES-256-GCM, stores the ciphertext, and strips the raw PII from the message. Agents receive structured context objects, never the raw personal data. Claude processes zero unencrypted PII.

Why: privacy enforced by architecture, not by policy — the AI physically cannot see what isn't passed to it.

🔗
Pattern 6 — Ecosystem MCP — Composable AI

The platform exposes a Model Context Protocol (MCP) server — a standard interface any external AI agent can call to browse restaurants, place orders, and track deliveries on behalf of a user. The same ordering logic, safety checks, and payment flows apply regardless of which AI triggered the request.

Why: makes the platform a reusable capability inside the broader AI ecosystem, not just a standalone app.

🧪
Pattern 7 — Quality Assurance Eval Framework — Testing AI Behaviour, Not Just Code

Unit tests verify that code runs correctly. They do not verify that an AI agent behaves correctly — a test that mocks the Claude API will pass even if the real model refuses, hallucinates, or misreads the state machine. DeliverOS has a separate eval suite that sends real prompts to real models and checks the outputs against expected behaviours.

58 eval cases across 6 agents: state machine correctness (does the agent advance the order to the right next state?), hallucination detection (does the judge catch invented restaurant names?), order flow edge cases, payment failure paths, safety (does the agent refuse PII requests?), and driver and merchant conversation flows. Evals run as part of the pre-deploy checklist — a regression in agent reasoning fails the suite before any code ships.
